# Downstream update ledger (architect / analyst / deployer)

Purpose: the per-module blueprint is the contract the customer ARCHITECT refines, the
ANALYST turns into a delta spec versus the customer's existing Semantius model, and the
DEPLOYER provisions into a tenant. Whenever a model-map or emitter change alters what the
blueprint carries, those downstream consumers must adapt. This file is the running ledger
of required downstream changes, accumulated across the model-map rework plans (see
[model-map-rework-plan.md](model-map-rework-plan.md)). It is consumed when we extend the
architect and analyst, which is the phase AFTER the model is coherent.

Notes:
- The permission CODES (baseline read/manage/admin, workflow gates, overrides) are emitted in
  section 8 and consumed by the deployer; the baseline / per-entity-tier story is settled (row 1).
  BUT plan-4 made the gate AND override codes for EMBEDDED entities deployment-dependent: a unit
  carrying an entity whose canonical realizing module is absent re-prefixes both to the installing
  unit (rows 6 / 11). That IS a real downstream change (the code is deployment-dependent and must be
  reconciled on master-install), not the no-op the original note assumed.
- Plan 5 (entity-scoped gate / override identity, design only, not executed) would retire the
  re-prefix in rows 6 / 11: a gate / override would be `<entity>:<verb>`, byte-identical whether
  emitted by a starter or by the canonical module, so master-install becomes an idempotent merge
  with no rename and no grant migration. If Plan 5 lands, the deployer reconciliation obligation in
  rows 6 / 11 disappears.
- Consulted / Informed RACI need NO new permission tier (resolved). Realization by layer:
  Responsible/Accountable -> permissions; Consulted -> a consultation lifecycle state (analyst
  provisions the state plus a read/comment grant) or an advisory read; Informed -> a
  notification side effect (a notify tool with operation_kind=side_effect, or a trigger_event;
  the deployer wires the notify action). `process_raci` stores the assignment with a
  POLYMORPHIC actor (persona via actor_role_id OR agent skill via actor_skill_id), restoring
  the AI-native twist (R can be an agent while A is a human). The derivation fans out to the
  permission, lifecycle-state, and notification layers, branching on persona-vs-skill for R.
  See [model-map-rework-plan.md](model-map-rework-plan.md).
- Row 10 (written up as [plan-4](plan-4-presence-conditional-required.md)) is the relationship-layer completion of Plan 3's M9 self-containment audit. M9 enforces self-containment at the ENTITY layer (`domain_module_data_objects` role / necessity); it does NOT inspect `data_object_relationships.is_required` edges, so a module can pass M9 yet still be non-deployable when an embedded entity has a required FK to a non-embedded entity. HIRING-STARTER is the worked case: it passes M9 (all rows `embedded_master` + the exempt `users` consumer) yet is non-deployable because `job_offers` has required edges to non-embedded `onboarding_journeys` / `offer_versions` / `background_checks`. The fix: `is_required` means "mandatory FK when the target is installed", never "force the target to install". A live scan found 531 `is_required=true` edges; 143 point at an entity that is `necessity=optional` in at least one module and 19 at one optional in every module it appears (e.g. `users --grants_consent_in--> gdpr_consent_records`). The starter kit is the most visible consumer; the same fix makes every module's M9 guarantee actually hold. The "closure" framing is gone everywhere (dropped with Plan 3 M9).
