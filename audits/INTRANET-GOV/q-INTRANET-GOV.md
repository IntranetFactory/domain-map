# Intranet Governance (INTRANET-GOV): questions waiting for you

## What this domain is
Know what is in every intranet, who owns it, and whether it is still true, without replacing the intranet you have.

Map every page, hub, library, and form across the intranets you already run, then make someone accountable for each one. Assign owners and review cadences, run recertification cycles so content gets confirmed current or flagged stale, and turn what you find into a prioritized improvement backlog. Connect SharePoint, Viva, Staffbase, Simpplr, or WordPress as governed sources and govern them from one place. It is an accountability layer over your existing intranets, not another intranet to migrate to.

> Grounding: this domain was built as a deliberately-thin white-space MVP (no pure-play vendor occupies the agnostic-overlay market; build study at `.tmp_deploy/INTRANET-GOV-phase0-2026-06-10.md`). The structural audit passed; the questions below are all "how deep do you want to build it", grounded in a fresh market-surface study of the adjacent flagship substrate (`.tmp_deploy/INTRANET-GOV-market-surface-2026-06-10.md`). None of these block the domain from being usable as-is. Note on solutions: all six are tagged "partial" on purpose (Orchestry, Powell, AvePoint are M365-bound, Siteimprove/Acquia are accessibility specialists, none is a clean primary for this agnostic market) - I did NOT mark any "primary", because the build evidence says there is no pure-play to promote.

q1: (answer this first) The domain stores per-content recertification records but nothing that DRIVES recertification on a schedule across many pages at once. Should I add the engine that runs the cycles?

- a) Add two things to the existing Governance module: a governance policy (the rules: how often to review, what counts as stale, what to archive, which content it applies to) and a recertification campaign (a scheduled batch that asks every owner to confirm their content and tracks who has and has not responded).
- b) Add both of those AND split the Governance module in two: one module for the policy/campaign setup, one for the day-to-day attesting and fixing.
- c) Leave it as-is: owners attest content one page at a time, with no scheduled cycle.

Recommended: a. Running recertification on a cadence is the domain's headline promise, and every adjacent tool drives it from exactly this pair: Microsoft's SharePoint Advanced Management ships a "Site Attestation Policy" that recurs on a schedule, Orchestry runs "Review Recurrence" campaigns, Simpplr enforces per-page validation windows, ShareGate runs recurring access reviews. Today we have the attestation record but nothing that schedules or batches it, so the cadence has to be run by hand. Choice (b) is how the bigger vendors eventually package it, but the split only pays off once policies, campaigns, and findings all exist - premature now, so add the entities first (a) and revisit the split later. Choice (c) keeps the MVP honest but means no scheduled cycle, which undercuts the main reason to buy this. This decision gates q2 and q3.

q2: Right now "stale" or "orphaned" content is only a number on a health score. Should each problem become its own trackable record (what is wrong, how bad, which page, who is fixing it, is it resolved)?

- a) Add a governance findings record for the content problems we already detect (stale, orphaned, inactive), and make the health score a roll-up of those findings.
- b) Same as (a), and also track permissions / oversharing problems (who can see content that should be restricted).
- c) Keep it as a score only, no per-problem record.

Recommended: a. Five of the six tools surveyed make findings a first-class record you can assign and close (Orchestry, SharePoint Advanced Management, ShareGate, Simpplr, Acquia all do; the score is treated as a view over the findings, not a replacement). A bare score tells you something is wrong but gives the workflow nothing to route or close. Start with the content-quality findings we already detect (a); oversharing/permissions (b) is real but overlaps with collaboration-governance tooling and can wait. This depends on q1 - findings are generated by the policies in (a).

q3: The Accessibility Governance capability is currently just a scan. Should accessibility become a tracked, standards-anchored record (which standard, what level, pass or fail, with an audit trail)?

- a) Add an accessibility conformance record AND register the accessibility standards (WCAG 2.2, EN 301 549, the European Accessibility Act) so content can be governed against them. (I already linked ADA and Section 508 to this domain this pass.)
- b) Just register the standards, keep accessibility as a scan only (no conformance record).
- c) Neither - leave accessibility as a scan with no standards or conformance tracking.

Recommended: a. Both accessibility specialists in this space (Siteimprove, Acquia Optimize) model conformance as a first-class, audited record evaluated against WCAG / EN 301 549, and the European Accessibility Act has been enforceable since June 2025 - so "we scanned it once" is no longer the same as "we can show conformance over time". If accessibility stays a capability of this domain, it should carry the standards and a conformance record. Pick (b) if you want the regulations on file but are not ready to model conformance as a tracked entity; (c) only if accessibility governance is out of scope (in which case the INTGOV-A11Y capability itself should be reconsidered).

## Optional (will not hold up the build)

q4: Improvements found during governance currently become a generic work item handed to Work Management. Should there be a dedicated, typed improvement/remediation record instead (linking a finding to its fix to the content it cleared)? (yes/no)

Recommended: yes, but only after q2. The build originally proposed a typed improvement record and dropped it in favor of the generic work-item handoff (which works fine today). A typed record pays off once findings (q2) exist, because then you can trace finding -> remediation -> recertified. Acquia and Siteimprove both model typed remediation requests. Non-blocking; the generic handoff is fine until then.
